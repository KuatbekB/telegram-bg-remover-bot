# ФонOFF — Telegram background remover

Telegram-бот принимает фотографию или изображение файлом и возвращает PNG с прозрачным фоном. Если первая модель ошиблась, пользователь может один раз запустить альтернативную обработку кнопкой «Попробовать ещё раз».

## Возможности

- фото и документы PNG, JPG, JPEG, WebP;
- сохранение результата без сжатия Telegram;
- две модели удаления фона: `u2net` и `isnet-general-use`;
- проверка подписки на канал (опционально);
- многопользовательская очередь с ограниченным числом ML-воркеров;
- лимиты размера входа и результата, защита от спама и перегрузки;
- повтор Telegram-запросов при временных сетевых ошибках и rate limit;
- команды `/privacy` и `/delete_me`, автоматическое удаление временных метаданных;
- отсутствие Telegram ID в прикладных логах;
- прогресс обработки и понятные сообщения об ошибках;
- Docker, readiness/heartbeat healthcheck и запуск от непривилегированного пользователя;
- автоматические тесты обработки и ограничений нагрузки.

## Локальный запуск

Требуется Python 3.12.

```powershell
cd D:\telegram-bg-remover-bot
py -3.12 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
$env:BOT_TOKEN="ТОКЕН_ОТ_BOTFATHER"
python bot.py
```

При первом использовании каждой модели `rembg` скачает её в каталог `~/.u2net`.

## Настройка

Все параметры задаются переменными окружения. Пример находится в `.env.example`.

| Переменная | Значение по умолчанию | Назначение |
|---|---:|---|
| `BOT_NAME` | `ФонOFF` | Название в приветствии |
| `REMBG_MODEL` | `u2net` | Основная модель |
| `RETRY_REMBG_MODEL` | `isnet-general-use` | Модель повторной попытки |
| `PRELOAD_MODELS` | `false` | Загрузить обе модели до приёма сообщений |
| `PRELOAD_PRIMARY_MODEL` | `true` | Загрузить основную модель до приёма сообщений; сокращает задержку первого запроса |
| `OMP_NUM_THREADS` | `2` | Число CPU-потоков ONNX/OpenMP на процесс |
| `MAX_DOWNLOAD_BYTES` | `20971520` | Максимальный размер файла, 20 МБ |
| `MAX_OUTPUT_BYTES` | `51380224` | Максимальный размер итогового PNG, 49 МБ |
| `MAX_IMAGE_PIXELS` | `12000000` | Максимальное количество пикселей изображения |
| `MAX_CONCURRENT_JOBS` | `2` | Максимум одновременных обработок |
| `MAX_QUEUE_SIZE` | `20` | Максимум ожидающих заданий сверх работающих |
| `UPDATE_CONCURRENCY` | `32` | Сколько обычных Telegram updates можно обрабатывать параллельно |
| `USER_COOLDOWN_SECONDS` | `3` | Пауза между запросами пользователя |
| `USER_RATE_LIMIT_COUNT` | `5` | Максимум принятых заданий одного пользователя за окно времени |
| `USER_RATE_LIMIT_WINDOW_SECONDS` | `60` | Размер окна пользовательского rate limit в секундах |
| `PROCESSING_TIMEOUT_SECONDS` | `120` | Мягкий таймаут одной ML-обработки |
| `TELEGRAM_MAX_RETRIES` | `2` | Число повторов важных Telegram-запросов |
| `RETRY_DATA_TTL_SECONDS` | `3600` | Срок хранения `file_id` для кнопки повторной обработки |
| `PRIVACY_CLEANUP_INTERVAL_SECONDS` | `60` | Интервал автоматической очистки временных метаданных |
| `REMOVE_WHITE_REMNANTS` | `false` | Глобальное удаление почти белых пикселей; обычно оставлять выключенным |
| `EDGE_FEATHER_RADIUS` | `0.4` | Небольшое смягчение края основной обработки |
| `SAVE_DEBUG_IMAGES` | `false` | Должно оставаться `false`: запуск с незашифрованным сохранением изображений заблокирован |
| `PRIVACY_CONTACT` | `@superski9` | Доступный пользователю контакт оператора бота |
| `PRIVACY_POLICY_URL` | политика Telegram | HTTPS-ссылка на применяемую политику конфиденциальности |
| `INSTANCE_LOCK_FILE` | системный временный каталог | Файл блокировки, запрещающий второй процесс бота |
| `READY_FILE` | системный временный каталог | Файл готовности процесса для healthcheck |
| `HEARTBEAT_FILE` | системный временный каталог | Периодически обновляемый файл активности event loop |
| `REQUIRE_CHANNEL_SUBSCRIPTION` | `false` | Требовать подписку на канал |
| `REQUIRED_CHANNEL` | `@superski9` | Канал для проверки подписки |
| `REQUIRED_CHANNEL_URL` | `https://t.me/superski9` | Ссылка на канал |

Если включена проверка подписки, бот должен быть администратором указанного канала.

## Docker

```powershell
docker build -t telegram-bg-remover-bot .
docker run --name fonoff-bot --restart unless-stopped `
  -e BOT_TOKEN="ТОКЕН_ОТ_BOTFATHER" `
  -e MAX_CONCURRENT_JOBS=2 `
  telegram-bg-remover-bot
```

Образ заранее скачивает обе стандартные модели. Состояние контейнера можно проверить командой:

```powershell
docker inspect --format='{{.State.Health.Status}}' fonoff-bot
```

Контейнер становится healthy только после загрузки основной модели, запуска очереди и Telegram-инициализации. Heartbeat также обнаруживает зависший event loop.

## Как работает многопользовательская обработка

Telegram-команды, сообщения и callback-кнопки принимаются параллельно. Изображения складываются в ограниченную очередь, а тяжёлую обработку выполняют только `MAX_CONCURRENT_JOBS` воркеров. Поэтому несколько пользователей могут пользоваться ботом одновременно, но внезапный поток файлов не запускает неограниченное число ONNX-задач и не съедает всю память.

Для каждого пользователя разрешено только одно активное или ожидающее задание. Переполненная очередь отклоняет новые изображения с понятным сообщением. Очередь хранится в памяти: при перезапуске незавершённые задания теряются, и пользователю нужно отправить файл повторно.

## Конфиденциальность

- изображения обрабатываются локальной моделью и обычно существуют только в оперативной памяти;
- `file_id`, необходимый для кнопки повторной обработки, хранится до `RETRY_DATA_TTL_SECONDS`; уже запущенное задание может удерживать его до завершения;
- метаданные rate limit автоматически очищаются, Telegram ID не записывается в прикладные логи;
- `/privacy` показывает пользователю актуальное описание обработки данных;
- `/delete_me` удаляет временные данные пользователя; если обработка уже идёт, окончательная очистка выполняется сразу после её завершения;
- `SAVE_DEBUG_IMAGES=true` намеренно блокирует запуск, поскольку проект не реализует шифрование пользовательских изображений на диске.

Перед публичным запуском укажите реальный `PRIVACY_CONTACT`. Если используется собственная политика вместо стандартной политики Telegram, разместите её по HTTPS, задайте `PRIVACY_POLICY_URL` и зарегистрируйте ссылку через BotFather. Для обработки и хранения данных пользователей Казахстана выбирайте подходящий сервер и проверяйте применимые требования с профильным специалистом.

## Рекомендации по ресурсам

- 2 CPU / 4 ГБ RAM: `MAX_CONCURRENT_JOBS=1`;
- 4 CPU / 8 ГБ RAM: `MAX_CONCURRENT_JOBS=2`;
- начальная публичная конфигурация: `MAX_QUEUE_SIZE=20`, `MAX_IMAGE_PIXELS=12000000`, `OMP_NUM_THREADS=2`;
- увеличивать `MAX_CONCURRENT_JOBS` стоит только после наблюдения за пиковым RSS, временем обработки и длиной очереди.

Токен бота нельзя добавлять в Git, Dockerfile или исходный код. Передавайте его только через переменную окружения `BOT_TOKEN`.

Пользовательские изображения обрабатываются только в памяти. Незашифрованное debug-сохранение реальных пользовательских изображений отключено.

На одном сервере разрешён только один процесс бота: второй экземпляр завершится до подключения к Telegram.
Текущая конфигурация рассчитана на массовое использование в пределах одного сервера. Для горизонтального масштабирования на несколько серверов потребуется webhook/API-слой, внешняя очередь, общее хранилище пользовательского состояния и распределённый лимитер (например, Redis).
